3.XSS

XSS这方面想要说出点名堂就要结合CSRF并进行区分

三种XSS

  • 存储XSS

    直接把恶意脚本存到数据库 这是最危险的

  • 反射XSS

    最普遍，一次性攻击，XSS代码出现在URL中，作为输入提交到服务器端，服务器端解析后响应，XSS代码随响应内容一起传回给浏览器，最后浏览器解析执行XSS代码

  • DOMXSS

    和反射的区别就是不经过服务器，DOM动态处理输入 恶意代码不出现在 HTML 源码中，而是在浏览器解析 DOM 时动态执行；

常见防护：

  • HTTPonly

    如果Cookie中设置了HttpOnly属性，那么通过JS脚本将无法读取到Cookie信息，这样能有效防止XSS攻击，窃取Cookie内容。 但是还是存在xss跨站语句，阻止的仅仅是获取cookie

    但这仅仅保护了cookie，跨站脚本能做的不仅是拿到用户的账号，能通过手法欺骗用户输入敏感信息，其实就是完全控制用户的前端脚本，做前端控制台能做的所有事

  • 字符过滤

    • 空格 用/代替<img/src="x"/onerror=alert("xss");>
    • 过滤关键词
      • 大小写
      • 双写
      • 拼接 <img src="x" onerror="a=aler;b=t;c='(xss);';eval(a+b+c)"> <script>top["al"+"ert"](xss);</script>
      • 编码
    • 引号 html标签中，可以不用引号。在js中，我们可以用反引号代替单双引号<img src="x" onerror=alert(xss);>
    • 括号 throw绕过onerror=eval;throw'=alert\x281\x29';

4.文件上传

文件上传比较基础 通常就是考几个绕过

  • 前端绕过：修改Burp请求或禁用JS上传webshell。
  • Content-Type/MIME类型绕过：将image/jpeg改为application/php。
  • 文件头绕过：在webshell文件前添加GIF文件头GIF89a。
  • .htaccess利用：上传自定义.htaccess文件，将特定扩展名（如.jpg）解析为PHP。
  • 日志包含：通过包含Apache/Nginx的访问日志文件来getshell。

5. SSRF

SSRF能做到的通常就利用几种协议进行渗透和攻击

  • file协议，这个很普通，就是阅读这台主机的文件

    http://example.com/ssrf.php?url=file:///C:/Windows/win.ini

  • dict协议，用于探测端口或者攻击redis

    dict://ip:port探测ip或找到ip探测内网端口

    gopher://ip:port/_TCP/IP数据流

    1.gopher协议数据流中，url编码使用%0d%0a替换字符串中的回车换行

生成器（Generator）是 Python 中一种特殊的迭代器，它可以通过简单的函数和表达式来创建。生成器的主要特点是能够逐个产生值，并且在每次生成值后保留当前的状态，以便下次调用时可以继续生成值。

生成器相关属性：

每当 Python 解释器执行一个函数或方法时，都会创建一个新的栈帧，用于存储该函数或方法的局部变量、参数、返回地址以及其他执行相关的信息。这些栈帧会按照调用顺序被组织成一个栈，称为调用栈。

​ 我最后是

类型 时机 访问的帧是谁 能不能逃出沙盒
普通生成器 (x for x in [1]) 生成器定义完后才执行 访问生成器自己的执行帧（gi_frame） ❌ 只能看到沙盒局部帧
自引用生成器 (a.gi_frame for i in [1]) 生成器定义过程中访问 a.gi_frame 拿到“正在创建它”的外部帧（上层帧） ✅ 能沿 f_back 爬到沙盒外部

一、 普通生成器： 生成器是在沙盒里“直接创建”的，只用到沙盒内的解析器，跳不出去

二、自引用生成器：帧链“拐出去” （得到一个过程的帧桟）

这里的关键点在于：

  • 当 Python 解释器在创建这个生成器表达式对象时，它要去“解析” a.gi_frame。
  • 但此时 a 变量指向的还没完全构造好的生成器对象本身。！！！！
  • 于是这个生成器对象的 frame 的局部变量环境里出现了一个对“自己”的引用，也就是说它捕获了创建它的那个上层帧对象。

现在它的“执行帧链”就变成：

这时 a.gi_frame.f_back 就是外部解释器的执行帧。
如果这个外部帧不是沙盒的局部帧（比如沙盒环境创建时的封装逻辑），继续 .f_back.f_back 多几层，就能摸到宿主 Python 的全局环境（即沙盒外）。

总结（通俗版一句话）

执行帧对象（frame） 是 Python 运行每个函数/表达式时自动生成的“运行记录本”，它知道自己是被谁调用的。

普通生成器在沙盒里只看到自己；
自引用生成器让你拿到了自己的“运行记录本（gi_frame）”；
你顺着这个记录本上的“上一页（f_back）”一路翻回去，就翻到了沙盒外的解释器世界。
