Fastjson反序列化

Fastjson反序列化

介绍

Fastjson（由阿里巴巴开源的java序列化工具），用于 Java 对象和 JSON 字符串之间的互相转换。
假设您有一个 Java 类 User。

  • 序列化： new User(...) \rightarrow {"name":"Alice", "age":25}//将java对象序列化成json，xml，二进制流等来方便传输和存储
  • 反序列化： {"name":"Alice", "age":25} \rightarrow new User(...)

简单的一个使用场景：

定义User类

序列化（Java 对象 → JSON 字符串）

反序列化（JSON 字符串 → Java 对象）

漏洞原理

FastJson的反序列化引入灵活的特性：@type 特性（AutoType）
现在AutoTypeSupport 特性默认是禁用的。
当FastJson解析包含@type的JSON字符时，会根据它指定的类名动态加载并创建实例

Fastjson看到这个@type就会动态加载User类并创建User对象

Fastjson反序列化漏洞就是让type加载Java生态中的特殊类，实现RCE

  1. 构造恶意 JSON：攻击者使@type指向一个在目标程序classpath中可被恶意利用的类

  2. 触发执行：让Fastjson尝试创建这个恶意类的实例，就会自动调用该类的methodgetter/setter/equals/hashcode

  3. RCE：攻击者触发JNDI注入，DNS解析，或调用

    等

    比如让 @type 指向能应发JNDI的类，攻击者构造JSON发起 （ldap://[Attack_IP]/Exploit），目标机反序列化这个Json的时候就发起了连接

利用和复现

我们先整理一下思路 就是用@type指向存在的类，然后执行恶意操作
但实际黑盒操作不可能知道有哪些类，所以可以用一些内置类

这里解释一下为什么要用不一样的Json payload：
更新的补丁 通过将存在安全风险的Class全路径的Hash值存储在黑名单中的方式进行校验，在反序列化过程中，Fastjson会检查 @type 字段指定的Class的Hash值是否存在于黑名单中。如果存在于黑名单中，Fastjson将拒绝实例化该Class，并抛出异常，从而防止恶意攻击者执行未授权等高危操作。

这里用缓存机制进行绕过：

  • 用字段a加载内置java.lang.Class去强制JVM加载并缓存com.sun.rowset.JdbcRowSetImpl类
  • 绕过点： 当 Fastjson 处于安全模式时，它通常只会检查 Payload 顶层对象 的 @type 是否在黑名单中。通过先用一个 合法的、不危险的类（java.lang.Class） 去加载 Gadget 类，可以绕过某些版本的安全检查或启用被限制的功能。
  • 再用b加载JdbcRowSetImpl,此时用的是内部缓存 绕过了DefaultJSONParser的黑名单检测。
  • 总结手法就是用内置安全类“白名单掩护” 和 “强制缓存”

总结

  • 就是利用FastJson库的@type特性，然后调用内置恶意类com.sun.rowset.JdbcRowSetImpl进行类似Log4j的JNDI注入，强迫服务器访问恶意服务器的恶意文件。
  • 其中有一次黑名单，但也可以用缓存机制，提前利用无害的class加载恶意类来写入缓存，绕过黑名单检测。
  • 这个漏洞的反序列化体现在Json序列化为java对象的过程中type属性调用了其他类

其他危险类可以通过网络查找，因为是黑盒，所以要么使用组件中的类，要么找内置类（可用于验证漏洞存在）
