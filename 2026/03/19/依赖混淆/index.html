依赖混淆.md

依赖混淆于2021年初发现，较为新颖。再一篇技术src博客中看到 这里学习并做一下记录（理论复现）

是一种高危的软件供应链攻击技术，核心在于利用现代包管理器（如 npm、pip、RubyGems 等）在解析和下载依赖项时的设计缺陷或配置不当，欺骗系统去公共代码库下载攻击者伪造的恶意包，（原本是从企业内部的私有代码库下载正确的内部包）

参考文章

漏洞成因

本质就是替换掉正常包为攻击者的恶意包而达到RCE。

利用包管理器设计时的默认行为与企业混合使用公私仓库时产生的信任边界模糊
解释一下就是以下两种情况

  1. 混合数据源解析
    有些企业开发环境或CI/CD 流水线配置多个源（既有内部的私有 Registry，也有公共的 Public Registry）。包管理器如果没有指定下载源可能从公共源下载
  2. 版本号劫持
    包管理器（如 npm 或 pip）下载时未锁定特定版本号或哈希的情况下，默认倾向下载最新版；
    所以攻击者在公共仓库发布同名改版本包，就提高了恶意包的优先级（例如设定为 99.9.9）

早期pip没有提供”仅从特定源安装特定包”的原生机制，且是合并私有和公有源，选择高版本
npm是先检查本地缓存，然后查询配置的registry（可能多个），最后选择符合版本约束的最高版本

攻击链

1.包名侦查与获取

获取方式主要如下几种

  1. 公开代码库：一种直接的来源，有些公司会开源部分源码，可能存在package.json、requirements.txt或pom.xml，这些里面直接引用了大量内部包名
    下图就是某次测试中找到的Nuget仓库

  2. js source map，前端错误监控： 打包后的JavaScript文件如果包含source map，或者错误堆栈被发送到公开的监控平台，内部依赖的名称就可能暴露。

  3. 其他蛛丝马迹： 例如github，技术博客，debug模式的错误信息和日志等，但收集难度较高

还有些情况 可以根据线索做命名规律推测，然后批量注册
（例如目标公司名为Acme，他可以合理推测存在acme-utils、acme-core、acme-common等包名）

2. 恶意包制作

制作高版本（999.0.0）同名包，包内容需要是安装即执行，因此，恶意代码必须挂载在包管理器的生命周期钩子（Lifecycle Hooks） 上，确保目标只要敲下 npm install 或 pip install，恶意代码就会立即在后台静默运行。
如果一个公司的 package.json 长这样：

开发人员执行： npm install，就回去一个个下载包
同时 攻击者npm publish了一个company-internal-pay

当开发者的电脑下载并准备“安装”这个包时，npm 引擎会自动扫描 scripts 里的生命周期钩子：

  • preinstall：在解压包后、正式安装前执行。
  • postinstall：在安装完成后立即执行。

3. 代码触发

恶意包发布后，攻击者只需等待。触发可能发生在多种场景：开发者本地运行npm install更新依赖、CI/CD系统执行定时构建、新员工克隆项目并安装依赖、依赖缓存过期后的重新拉取。

值得注意的是，即使企业的生产构建严格锁定了依赖版本，开发环境和CI/CD的”测试分支”往往没有同等严格的控制。攻击者可能首先获得的是开发者的本地机器或测试环境的访问权限，然后以此为跳板进一步渗透

4. 外带与持久化

只需要带外出 API密钥，数据库凭证，SSH私钥，AKSK等环境变量/配置文件，就能尝试构建持久化访问，留后门，扩大污染

案例

实际的赏金漏洞中，难以证明影响
站点redacted.com, BP的插件JS Miner中，依赖项中发现了非公共库的包名
recovery-npm-package

怀疑可能存在 依赖混淆（dependency confusion） 的情况。应用似乎引用了一个未公开注册的内部依赖。如果构建过程从公共 npm 注册表解析依赖项，攻击者可能注册同名包并将恶意代码注入构建管道。
正确注册了它，并准备了一个更受控的有效载荷，以捕获清晰的执行证据。

包中包含两个文件：一个简单的 index.js 和一个包含 preinstall 钩子 的 package.json。

package.json 内容如下：

如果任何系统安装这个包，preinstall 脚本会自动执行，并将 /etc/passwd 内容连同执行安装的主机名发送到我的 Burp Collaborator 服务器。
几分钟内，我的 Burp Collaborator 仪表盘开始接收到交互请求。起初只是一个请求，随后又一个，然后是更多请求。

每个交互都包含 HTTP 请求，内含 /etc/passwd 内容，以及执行系统的主机名和元数据。多个 IP 地址访问我的 Collaborator 域。有效载荷已成功执行，这确认了恶意包已在某处安装，preinstall 钩子在安装过程中执行。
到这就证明了依赖混淆攻击的RCE
但甲方说：
回调的 IP 是中国的云服务器，不是目标公司的内网 CI/CD 系统。
公司内部项目用了 YarnLock 文件（锁死了真实依赖的哈希和来源），所以恶意的同名包根本进不去核心构建系统。

那些触发 RCE 的回调，其实是互联网上自动监控 npm 镜像源的第三方安全扫描器或爬虫。只要有人发新包，它们就会自动拉取分析，从而踩中了作者的陷阱。

防御手段

具体的手段就不记了，引用的文章中有记录，这里就提一下大方向

  1. 安装文件中明确标定版本号
  2. 明确只从私有仓库下载
  3. 占位包策略，预先注册同名包
